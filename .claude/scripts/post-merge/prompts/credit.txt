# What you are evaluating

A pull request was just merged. Below are the PR's metadata,
description, comments, file list, and any linked issue (with that
issue's body and comments). Use them to decide who, if anyone, should
be credited for this work in the Acknowledgments section.

# Decision rubric

For each candidate contributor (PR author, issue author, issue
commenters, PR reviewers/commenters), ask:

1. Did they author the merged code? (PR author of an external PR)
2. Did they provide a concrete fix, workaround, working patch,
   detailed root-cause analysis, or reference diff that was directly
   used in the merged change? (issue / comment contributors)
3. Did they file a high-signal report — reproducible repro steps,
   logs, bisection — that materially shortened the path to the fix?
   (issue authors whose report directly enabled the diagnosis)

Weak signals that DO NOT warrant a credit on their own:
- "Me too" / "+1" comments without new technical content.
- Reporting a bug without diagnosis or repro steps.
- Asking clarifying questions.
- Being pinged by another commenter.

If the PR is authored by `aaddrick` (the project maintainer) AND no
external commenter provided substantive technical input, the typical
output is `should_update: false` with `entries: []`.

If a bot authored the PR (`dependabot[bot]`, `renovate[bot]`,
`github-actions[bot]`), output `should_update: false` with
`entries: []`.

# Choosing the action

For each contributor you decide to credit:

- If they are NOT in <current_acknowledgments> at all → action `add`.
  The `description` field is the body of the new one-liner that will
  appear AFTER the linked username, e.g.:
    "for fixing the cowork session-init regression on Claude
    Desktop 1.5354.0 (#553)"
  Do NOT include their @handle, the bracketed link, or the leading
  "for" + bold name in the description — only the trailing prose.

- If they ARE in <current_acknowledgments> as a one-line entry →
  action `promote`. The `description` field is the NEW bullet only;
  the existing one-line description is preserved automatically as the
  first bullet of the new block.

- If they ARE in <current_acknowledgments> as a multi-bullet entry →
  action `append`. The `description` field is the new bullet text.

Never include a leading "- " or "  - " in the description. The
applier inserts the bullet markup and indentation.

# Wording guidance

Match the prevailing voice of existing entries: terse, technical,
specific. State the bug or surface affected and the nature of the
fix or analysis. Reference issue/PR numbers in parentheses when the
linkage to a specific issue would help future readers, e.g.
"(#421)", "(#553)". Avoid effusive language ("amazing", "fantastic"),
generic praise ("great work"), or filler ("contributed by helping
out"). Aim for the tone of:

  "for fixing the duplicate tray icon on OS theme change with an
  in-place setImage/setContextMenu fast-path that avoids the KDE
  Plasma SNI re-registration race"

  "for diagnosing the Cowork CLAUDE_CODE_OAUTH_TOKEN env-strip bug
  with a working reference diff"
